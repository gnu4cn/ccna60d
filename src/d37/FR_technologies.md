# 帧中继技术

帧中继属于一种非广播的多路访问（Non-Broadcast Multi-Access, NBMA）技术。这意味着除了使用点对点接口的情形，咱们必须处理地址解析的问题。

帧中继中的本地二层地址，称为数据链路连接标识符（DLCI），这种地址仅在本地有效。比如，在中心分支的环境中，中心设备就应有着与其各个分支通信的唯一 DLCI。（译注：即中心 FR 交换机上的 DLCI 不能重复。）

当咱们检查某一思科设备上的帧中继 PVC 状态时，咱们将看到某个由 LMI 定义的代码，可以是以下三者一：

- `Active`（一切正常）
- `Inactive`（本地节点上无问题，但远端节点上可能存在问题）
- `Deleted`（业务提供商网络中的问题）


作为一个实例，思科的设备就提供了三种 LMI 类型：

- `CISCO`
- `ANSI`
- `Q.933a`


思科路由器配置为自动尝试全部这三种 LMI 类型（从 `CISCO` LMI 开始），并使用与业务提供商所使用 LMI 匹配的类型，因此这点在设计阶段不应过多关注。

在设计阶段需要考虑的最重要方面之一，是所用到的地址解析方法。当咱们在咱们的设计中，使用了多点接口（即可终止多条二层电路的接口）时，咱们就需要找到某种提供三层到二层解析的方法。正如前面曾讨论过的，咱们有两个选项，可帮助咱们实现这一目的：

- 动态方式：利用反向 ARP 协议
- 静态方式：经由思科设备上的 `frame-relay map` 这条静态配置命令


**注意**：为了验证三层到二层的解析是否已成功，咱们可使用 `show frame-relay map` 这一命令。

在多点接口上，反向 ARP 会自动发生。在添加了 IP 地址到某一配置了帧中继的接口后，这一功能便被启用。此时，一些请求就会从指派到该特定接口的所有电路上，针对该接口正运行的任何受支持协议发出。

这一请求过程，可以 `no frame-relay inverse-arp` 这条命令禁用，但咱们绝不会设计一个将停止响应请求的网络。根据设计，反向 ARP 的响应就不能被禁用，进而运行帧中继的设备，将始终协助另一尝试经由帧中继的反向 ARP 完成三层到二层解析的设备。

帧中继设计中的这种反向 ARP 行为，将通过之前曾讨论过的 [复制单播](./NBMA.md) 方法，自动协助广播。在用到反向 ARP 时，广播支持默认存在。

当咱们使用物理接口连接两台路由器到帧中继云时，那么从帧中继视角看，这便意味着这两个特定接口便属于多点接口，因为物理的帧中继接口默认便是多点结构。即便两台路由器间的连接，可能看似点对点的，但其仍是帧中继的多点连接。

由于他们使用了多点接口，因此默认情况下这两台设备都将通过使用反向 ARP，动态地处理三层到二层的解析。当咱们希望设计其中不使用反向 ARP 的方案时，咱们可在各个设备上关闭这种动态映射行为，并随后配置静态的帧中继映射。

当涉及三层到二层的解析时，点对点的配置便是理想选择，因为在使用此类接口类型时，这一过程不会发生。在配置点对点的帧中继时，咱们会用到点对点子接口，而这些子接口不会像多点接口情形下那样，从 LMI 获取 DLCI 分配。

另一选项便是创建出子接口，并将他们声明为多点接口。这些接口类型，会完全像物理多点接口那样行事，因此咱们就需要确定出要用到的解析方法，要么是反向 ARP，要么是静态映射。可使用这两种方法的组合，例如，在连接的一端实施反向 ARP，而在另一端定义静态映射。

部分网状的设计和配置，将最具挑战性。这意味帧中继环境中所涉及的全部端点之间，都会配置二层电路。

在中心分支环境中，各分支未相互直连，因此这意味着他们无法经由反向 ARP 解析对方。为了解决这些问题，咱们可执行以下任一操作：

- 提供额外的静态映射
- 配置点对点子接口
- 设计中心分支的基础设施，以便三层的路由设计，能够解决解析问题（例如，通过使用 OSPF 的点对多点网络类型）

帧中继支持可影响服务质量（QoS）的标记。例如，帧中继头部包含一个 DE（Discard Eligible，可丢弃）位。在 QoS 的帧中继环境下，数据包可标记以这个 DE 位，而这会通知业务提供商，这些特定数据包并不很重要，而可在拥塞情形下被丢弃。这种行为将使那些没有设置这个 DE 位的数据包优先。

其他可在帧中继环境中配置的参数，分别是前向显式拥塞通知（Forward Explicit Congestion Notifications, FECN），及后向显式拥塞通知（Backwards Explicit Congestion Notifications, BECN），二者常会作为考试问题出现。当帧中继设备配置了要这样做时，那么他们便可通知设备拥塞，进而引起发送速率下降。
